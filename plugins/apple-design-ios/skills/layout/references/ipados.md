# iPadOS
Users can freely adjust window dimensions to a minimum width and height, mirroring macOS window behavior. When designing your layout, you must account for this full range of possible sizes and the associated resizing behavior. Refer to [iPadOS](multitasking.md#iPadOS) and [iPadOS](windows.md#iPadOS) for detailed guidance.

- **As someone resizes a window, defer switching to a compact view for as long as possible.** Prioritize designing for the full-screen experience, only transitioning to a compact view when the complete layout cannot fit. This approach maintains UI stability and predictability across different usage scenarios. For complex layouts, such as those utilizing [Split views](split-views.md), it is preferable to hide tertiary columns, like inspectors, as the view narrows.
- **Test your layout at common system-provided sizes, and provide smooth transitions.** Since window controls allow users to arrange windows into halves, thirds, or quadrants of the screen, your layout must be verified at each of these sizes across different devices. Ensure that UI adjustments between the minimum and maximum window dimensions are as seamless as possible, minimizing unexpected changes.
- **Consider a convertible tab bar for adaptive navigation.** Many applications do not require an exclusive choice between using a tab bar or a sidebar; instead, you can implement a tab bar style that supports both. The application initially launches in either a sidebar or tab bar mode, allowing users to switch between them via tap. As the view resizes, the presentation style adapts to fit the available width. For general guidance, see [Tab bars](tab-bars.md). Developer implementation details can be found in [sidebarAdaptable](apple:SwiftUI/TabViewStyle/sidebarAdaptable).
