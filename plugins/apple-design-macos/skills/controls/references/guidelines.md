# Controls — full guidelines
Users can add controls to Control Center by pressing and holding in an empty area of Control Center, to the Lock Screen via customization, or to the Action button by configuring it in the Settings app.

### Anatomy
Controls consist of a symbol image, a title, and an optional value. The symbol visually indicates the control's function and may be sourced from [SF Symbols](sf-symbols.md) or defined as a custom asset. The title provides context regarding the control's subject, while the value communicates its current state. For instance, the title might show a light's name, and the value might indicate its power status (on or off).

Controls present their information based on the viewing context:

- In Control Center, a control displays its symbol; additionally, it shows its title and value when viewed at larger sizes.
- On the Lock Screen, a control displays only its symbol.
- For iPhone devices utilizing a control assigned to the Action button, pressing and holding it displays the control’s symbol in the Dynamic Island, along with its value (if applicable).

### Best practices
- **Offer controls for actions that provide the most benefit without requiring users to launch your application.** For instance, launching a Live Activity from a control facilitates a smooth experience, keeping users informed about progress without needing to navigate into your app. For detailed information, consult [Live Activities](live-activities.md).
- **Update controls when a user interacts with them, when an action finishes, or remotely via a push notification.** Ensure the control's contents accurately reflect its current state and indicate if an action is ongoing.
- **Select a descriptive symbol that conveys the control's behavior.** Since controls may not display their title and value depending on where they are placed, the symbol must communicate sufficient information about the control's function. For control toggles, provide symbols for both the enabled and disabled states. For example, use SF Symbols `door.garage.open` and `door.garage.closed` to represent a control that controls a garage door. For guidance, see [SF Symbols](sf-symbols.md).
- **Use symbol animations to emphasize state changes.** For control toggles, animate the transition between both on and off states. For control buttons executing actions that take time, animate continuously while the action is in progress and cease animation upon completion. For developer guidance, see [Symbols](apple:Symbols) and [SymbolEffect](apple:Symbols/SymbolEffect).
- **Select a tint color that aligns with your app’s branding.** The system applies this tint color to the symbol of a control toggle when it is in its on state. Furthermore, when a user performs an action via the Action button, the system utilizes this tint color to display both the value and the symbol in the Dynamic Island. For guidance, see [Branding](branding.md).
- **Help users provide any additional information the system requires to execute an action.** A user might need to configure a control before performing a desired action—for example, selecting a specific light in a house to toggle its power. If configuration is necessary for a control, prompt users to complete this step upon initial addition. Users retain the ability to reconfigure the control at any time. For developer guidance, see [promptsForUserConfiguration()](apple:SwiftUI/ControlWidgetConfiguration/promptsForUserConfiguration()).
- **Provide hint text for the Action button.** When a user presses the Action button, the system displays hint text to clarify what occurs when they press and hold. When someone presses and holds the Action button, the system executes the action that has been configured for it. For developer guidance, see [controlWidgetActionHint(_:)](apple:SwiftUI/View/controlWidgetActionHint(_:)-5yoyh).
- **If your control title or value is variable, include a placeholder.** Placeholder information informs users about the function of your control when its title and value are context-dependent. The system displays this information when a user accesses the controls gallery in Control Center or the Lock Screen, or before they assign it to the Action button.
- **Hide sensitive information when the device is locked.** When the device is secured, consider having the system redact the title and value to conceal personal or security-related data. Specify whether the system must also redact the symbol state. If specified, the system redacts the title and value and displays the symbol in its off state.
- **Require authentication for actions that impact security.** For example, require users to unlock their device before they can access controls related to locking or unlocking a house door or starting a car. For developer guidance, see [IntentAuthenticationPolicy](apple:AppIntents/IntentAuthenticationPolicy).

### Camera experiences on a locked device
If your application supports camera capture, iOS 18 introduces a control that allows direct launch into your app’s camera experience even when the device is secured. However, any operation requiring functionality beyond simple capture necessitates user authentication and unlocking the device to proceed within your application. Detailed guidance for developers is available at [LockedCameraCapture](apple:LockedCameraCapture).

- **Maintain consistency between your app's interface and the camera experience.** Utilizing a shared UI capitalizes on user familiarity with your application. When users capture content and subsequently select an option to perform secondary actions—such as editing a picture or publishing it to social media—the transition into the app will be fluid.
- **Include instructions on how to add this control.** Ensure users understand the process for implementing the control that initiates this camera experience.
