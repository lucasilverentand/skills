# Enhancing the accessibility of a chart
When you utilize Swift Charts to generate a visualization, the system provides a default implementation of [Audio graphs](apple:Accessibility/audio-graphs), alongside a standard accessibility element for every mark (or collection of marks) that describes its specific value.

**Consider using Audio Graphs to provide VoiceOver users with deeper insights into your chart.** You can modify the default implementation of Audio Graphs supplied by Swift Charts by supplying a chart title and a descriptive summary that VoiceOver reads aloud, enabling users to grasp the chart's purpose and primary features. If you choose not to use Audio Graphs, you must provide an overview of the chart's structure and intended function. For instance, you need to specify the chart type—such as line or bar—explain what each axis represents, and detail aspects like the upper and lower bounds of the axes.

> **Important**
> Unlike an image, which typically requires a single descriptive accessibility label, a chart often necessitates providing an accessibility label for every significant or interactive element. Depending on the chart's function and the density and scope of its marks, you must determine whether it is necessary to describe each mark individually or if describing groups of marks enhances the accessibility experience. In certain scenarios, providing a single, high-level summary label is appropriate, such as when using a small chart version within a button that reveals a more detailed view.

**Write accessibility labels that align with the intended purpose of your chart.** For example, Maps displays elevation data for a cycling route using a chart that visualizes the change in elevation across the journey. The goal of this chart is to convey a sense of the entire route's terrain, not to provide individual elevation points. For this reason, Maps offers accessibility labels that summarize elevation shifts over a segment of the route, rather than labeling every single moment. Conversely, Health provides an accessibility label for each bar in the Steps chart because its purpose is to convey the user's actual step count for that specific tracking period.

The following guidelines will assist you in composing effective accessibility labels for chart components.

- **Prioritize clarity and comprehensiveness.** Generally, simply stating a data value is insufficient unless you also include context that helps users understand it, such as the associated date or location. Aim to concisely describe the context for a value without repeating information that is already available elsewhere, such as an axis name provided by Audio Graphs or your overview. Follow this context-setting information with a brief description of the element's specific details.
- **Avoid using subjective terms.** Words that convey interpretation—such as rapidly, gradually, or almost—should be avoided. To allow users to form their own conclusions, use the actual values in your descriptions.
- **Maximize clarity in data descriptions by avoiding potentially ambiguous formats and abbreviations.** For instance, stating "June 6" is clearer than using "6/6"; similarly, spelling out "60 minutes" or "60 meters" is clearer than using the abbreviation "60m."
- **Describe what the chart details represent, not how they appear.** Consider a chart that uses red and blue colors to visually differentiate two distinct data series. It is vital to create accessibility labels that identify what each series represents, but describing the colors used for visual representation can introduce irrelevant information and be distracting.
- **Maintain consistency throughout your application when referencing a specific axis.** For example, if you consistently mention the X-axis first, users will spend less time trying to determine which axis is relevant in a description.

**Hide visible text labels for axes and ticks from assistive technologies.** Axis and tick labels assist users in visually assessing trends and estimating mark values. Since VoiceOver users can obtain mark values and trend information through accessibility labels and Audio Graphs, the content in the visible labels is generally not required.
