# Customizing pointers
- **Prefer system-provided pointer effects for custom elements that behave like standard elements.** When a custom element mimics the behavior of a native control, users naturally expect interaction cues consistent with standard operating system conventions. For instance, if buttons within a custom toolbar lack the typical highlight feedback, users may incorrectly assume they are malfunctioning.
- **Use pointer effects in consistent ways throughout your app.** If your application involves drawing, for example, apply a uniform pointer experience across all drawing canvases so users can transfer knowledge gained in one area to others.
- **Avoid creating gratuitous pointer and content effects.** Users are attuned to changes in the appearance of the pointer or the underlying UI element, and they expect these shifts to convey functional information. Effects that are purely decorative can distract or frustrate users without offering practical benefit.
- **Keep custom pointer shapes simple.** Ideally, the pointer's form should clearly indicate the action available in the current context without demanding excessive attention. If users cannot immediately interpret your custom pointer shape, they will likely spend time trying to determine its meaning.
- **Consider enhancing the pointer experience by displaying custom annotations that provide useful information.** You might, for example, display X and Y coordinates when a user holds the pointer over a graphing region. Keynote utilizes annotations to show the current width and height of an adjustable image.
- **Avoid displaying instructional text with a pointer.** A pointer that displays instructions can make an application appear overly complex or difficult to operate. Instead of providing guidance, prioritize interface clarity and simplicity so users can quickly understand how the app works, whether they are using the pointer or touching the screen.
- **Consider the interplay of shadow, scale, and element spacing when defining custom hover effects.** Generally, reserve scaling for elements that can increase in size without encroaching upon adjacent components. For example, scaling is ineffective for a table row because expansion would overlap neighboring rows. For an element with limited surrounding space, consider using a tint effect rather than scale and shadow. Note that employing a shadow without including scaling is often ineffective, as an unscaled element fails to convey the impression of approaching proximity, even if its shadow suggests elevation above the screen.
