# Best practices
**Consider each screen’s purpose**. Pay close attention to the sequence of elements on a screen, prioritizing the most critical information upfront. Format your text for maximum readability. If a single idea requires more than one screen to convey, consider breaking the content across multiple screens and mapping out the information flow between them.

- **Be action oriented.** Using active voice and explicit labels facilitates user navigation, whether moving between screens or progressing through steps within an app. When labeling buttons and links, it is generally most effective to use a verb. Prioritize clarity above all else; avoid overly clever or cute labels. For instance, "Send" is often superior to "Let’s do it!" Similarly, instead of using "Click here," use descriptive phrases like "Learn more about UX Writing" for links. This is particularly vital for users accessing the app via screen readers.
- **Build language patterns.** Consistency fosters familiarity, contributing to an app that feels cohesive, intuitive, and thoughtfully designed. Furthermore, it simplifies the writing process by allowing you to reuse established patterns repeatedly.
- **Adopt capitalization rules that align with your app’s style, then apply them consistently.** While specific components, such as **Content**, have defined guidelines, your text formatting reflects the app's overall voice. Generally, title case suggests a formal tone, whereas sentence case is more casual. Select a style for each UI element type and maintain it throughout the application—for example, using title case for all alerts or sentence case for all headlines.
- **Give clear guidance and use consistent language throughout processes with multiple steps.** If a flow spans several screens, define how actions that transition users between steps will be labeled. Begin flows with language like “Get Started.” You may use the button label to hint at the subsequent step, or employ terms like “Continue” or “Next,” but maintain consistency in your choice. Use language such as “Done” to signify the completion of a flow.
- **Use possessive pronouns sparingly.** Possessive pronouns like *my* and *your* often add unnecessary context. For example, “Favorites” conveys the same meaning as “Your Favorites,” but is more concise. If you must use possessive pronouns, apply them consistently throughout the app and avoid shifting perspectives. Avoid using *we* entirely, as it can be ambiguous regarding who the "we" refers to. This is especially true in error messages; instead of saying, “We’re having trouble loading this content,” use the clearer phrasing, “Unable to load content.”
- **Write for how people use each device.** Since users may interact with your app across different devices, while language consistency is required, consider where text adjustments would suit different hardware. Accurately describe gestures for each device—for example, using "tap" instead of "click" when referring to a touch-based device like iPhone or iPad.

The context, screen size, and location where a device is used all influence your writing approach. iPhone and Apple Watch, for instance, allow personalization opportunities but demand brevity due to their small screens. Conversely, TVs are often viewed in shared living spaces by multiple people, requiring consideration of the broader audience. Larger screens also necessitate brevity because text must be large enough to be viewed from a distance.

- **Provide clear next steps on any blank screens.** An empty state—such as a completed to-do list or bookmarks folder with no items—offers an opportunity to welcome users and educate them about your app. Empty states can also establish your app's voice, but ensure the content is relevant and fits the context. If an empty screen doesn't clearly indicate what to do next, it can be daunting; therefore, guide users on possible actions and provide a button or link to execute them. Remember that empty states are usually temporary, so do not display crucial information that might subsequently disappear.
- **Write clear error messages.** It is always preferable to help users prevent errors. When an error message becomes necessary, display it as close as possible to the problem location, avoid assigning blame, and clearly state how the issue can be resolved. For example, stating “Choose a password with at least 8 characters” is more helpful than saying, “That password is too short.” Remember that errors can be frustrating. Interjections like “oops!” or “uh-oh” are typically unnecessary and may sound insincere. If language alone cannot adequately address an error likely affecting many users, use that as a chance to rethink the interaction entirely.
- **Choose the right delivery method.** There are numerous ways to capture user attention, whether they are actively using your app or not. When you need to communicate something, weigh the message's urgency and importance. Consider the context in which the user might encounter the message, whether immediate action is required, and how much supplementary information they may need. Select the appropriate delivery method and match your tone to the situation. For guidance, refer to **Notifications**, **Alerts**, and **Action sheets**.
- **Keep settings labels clear and simple.** Help users locate necessary settings by labeling them as practically as possible. If the label alone is insufficient, include an explanation. Describe the setting's function when enabled; users can usually infer the opposite state. For example, in the Handwashing Timer setting for Apple Watch, the description explains that a timer can begin during hand washing. It is unnecessary to state that the timer will not start when this setting is disabled.

If you must direct a user to a specific setting, provide a direct link or button rather than attempting to describe its location. For guidance, see **Settings**.

**Show hints in text fields.** If your app requires users to input their own text, such as account or contact details, label all fields clearly and use hint or placeholder text to guide users on the required format. You may provide an example in the hint text, like “name@example.com,” or describe the required information, such as “Your name.” Display errors immediately adjacent to the field and instruct users on correct entry rather than criticizing incorrect input. “Use only letters for your name” is preferable to “Don’t use numbers or symbols.” Avoid robotic error messages lacking helpful information, such as “Invalid name.” For guidance, see **Text fields**.
