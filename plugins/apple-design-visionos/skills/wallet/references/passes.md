# Passes
- **Offer to add new passes to Wallet.** When a user action results in a new pass—such as checking into a flight, purchasing an event ticket, or enrolling in a store reward program—you can present system-provided UI to facilitate adding the pass to Wallet with a single tap (refer to [addPasses(_:withCompletionHandler:)](apple:PassKit/PKPassLibrary/addPasses(_:withCompletionHandler:))) for implementation guidance). If users wish to review a pass before adding it, you may display a custom view that presents the pass along with an Add to Apple Wallet button; see [PKAddPassesViewController](apple:PassKit/PKAddPassesViewController) for developer details.
- **Help people add a pass that they created outside of your app.** If users generate a pass via your website or another device, suggest adding it to Wallet the next time they launch your application. If users decline this suggestion, do not prompt them again.
- **Add related passes as a group.** If your application generates multiple passes, such as boarding passes for a multi-segment flight, include all passes simultaneously so users are not required to add each one individually. If your website allows users to receive a collection of passes—like event tickets—bundle them together so they can download all items at once. For developer guidance, consult [Distributing and updating a pass](apple:WalletPasses/distributing-and-updating-a-pass).
- **Display an Add to Apple Wallet button to let people add an existing pass that isn’t already in Wallet.** If users previously declined your suggestion to add a pass to Wallet—or if they have removed the pass—a dedicated button allows them to easily add it back if they change their mind. You can place an Add to Apple Wallet button wherever the corresponding pass information appears in your app. For developer guidance, see [PKAddPassButton](apple:PassKit/PKAddPassButton). Additionally, you may display an Add to Apple Wallet badge in email or on a webpage; refer to [Add to Apple Wallet guidelines](https://developer.apple.com/wallet/add-to-apple-wallet-guidelines/) for instructions.
- **Let people jump from your app to their pass in Wallet.** Wherever your application displays details about a pass that already resides in Wallet, you can provide a link to open it directly. Label this link appropriately, such as “View in Wallet.”
- **Tell the system when your pass expires.** Wallet automatically conceals expired passes to prevent clutter while still offering a button for users to revisit them. To assist the system in hiding passes accurately, ensure that you correctly set the expiration date, relevant date, and voided properties for each pass; see [Pass](apple:WalletPasses/Pass) for developer guidance.
- **Always get people’s permission before deleting a pass from Wallet.** For instance, you could implement an in-app setting allowing users to specify whether they wish to delete passes manually or have them removed automatically. If deletion is necessary, display an alert before proceeding.
- **Help the system suggest a pass when it’s contextually relevant.** Ideally, passes appear automatically when they are needed, eliminating the need for users to manually locate them. When you provide information regarding a pass's relevance in time and location, the system can display a link to it on the Lock Screen when users are most likely to want it. For example, a gym membership card might appear on the Lock Screen as members enter the facility. See [Showing a Pass on the Lock Screen](apple:WalletPasses/showing-a-pass-on-the-lock-screen) for developer guidance. Starting with iOS 18 and watchOS 11, the system initiates a Live Activity for poster event ticket style passes when they become relevant.
- **Update passes as needed.** While physical passes typically remain static, a digital pass can reflect changes to events. An airline boarding pass, for example, may automatically update to show flight delays or gate changes.
- **Use change messages only for updates to time-critical information.** Since a change message interrupts the user’s current workflow, it is crucial to send one only when you are providing an update they must know about. For example, users need notification of a gate change in a boarding pass, but they do not require notification if a customer service phone number changes. Never use a change message for marketing or other noncritical communications. Change messages are supported on a per-field basis; see [Adding a Web Service to Update Passes](apple:WalletPasses/adding-a-web-service-to-update-passes) for developer guidance.
