# Platform guidance — visionOS
By default, visionOS utilizes ARKit algorithms to manage features such as environment lighting, segmentation, matting, persistence, and world mapping. These algorithms operate continuously, allowing applications and games to automatically leverage ARKit capabilities while operating in the Shared Space.

ARKit does not transmit data to apps running in the Shared Space; therefore, accessing ARKit APIs requires your application to initiate a Full Space. Furthermore, specific features like Plane Estimation, Scene Reconstruction, Image Anchoring, and Hand Tracking require explicit user permission to access any underlying information. Refer to [Setting up access to ARKit data](apple:visionOS/setting-up-access-to-arkit-data) for developer guidance.

In visionOS, user input is inherently private. The system automatically generates hover effects when users view interactive elements you construct using SwiftUI or RealityKit, providing necessary visual feedback without revealing the user's focus point until they tap. See [Eyes](eyes.md) and [visionOS](gestures.md#visionOS) for more information.

Developer access to device cameras operates differently in visionOS compared to other platforms. Specifically, the back camera provides blank input and is offered solely as a compatibility feature; conversely, the front camera supplies input for [visionOS](shareplay.md#visionOS), but only after users grant permission. If the iOS or iPadOS application you are porting to visionOS includes a feature dependent on camera access, you must either remove it or replace it with an option allowing users to import content. For developer guidance, consult [Making your existing app compatible with visionOS](apple:visionOS/making-your-app-compatible-with-visionos).
