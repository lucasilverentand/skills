# Schema Layout

## Single source of truth

A single `packages/db/` workspace owns all schema definitions. Every app and service imports from `@repo/db`. No local schema copies. No ORM-per-service. Because schema drift across services is one of the worst forms of tech debt — it's silent until it causes a production bug.

## File structure

```
packages/db/
  schema/
    orders.ts        # order, line_items, order_history
    auth.ts          # users, sessions, accounts (Better Auth)
    billing.ts       # subscriptions, invoices, payments
    index.ts         # barrel re-export of all tables
  migrations/
    0001_initial.sql
    0002_add_billing.sql
```

- **Group by bounded context**, not by table. `schema/orders.ts` contains `orders`, `line_items`, and related tables — because they change together and belong to the same domain.
- **Barrel export** in `schema/index.ts` re-exports all tables. All imports go through `@repo/db`.

## Better Auth tables

The framework manages `users`, `sessions`, `accounts`, `verifications`. Extend with `tenant_id` and custom columns. Don't fight the framework's schema — add to it, don't replace it.

## Migrations

`packages/db/migrations/` contains committed SQL files. Generated by `drizzle-kit generate`, reviewed in PRs like any other code, applied by `drizzle migrate` in CI before deploy.

## Anti-patterns

- **Schema definitions scattered across services** — guarantees drift, duplication, and conflicting assumptions about column types.
- **Migrations in application code** — migrations belong in the DB package, not in the service that happens to run first.
- **Hand-written SQL migrations alongside Drizzle-generated ones** — pick one source of truth. Drizzle generates; you review and commit.
- **Importing from another service's schema** instead of from `@repo/db` — creates hidden coupling between services.
