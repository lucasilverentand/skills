# CLAUDE.md Structure

The root `CLAUDE.md` is the primary context file for Claude Code. It's loaded automatically when Claude opens the project.

## Section order

Organize by what agents need most urgently — commands first, context second, conventions last.

### 1. Project identity (2-3 lines)

What this project is and what it does. One sentence is ideal.

```markdown
# Project Name

A [type] that [does what] for [who]. Built with [key tech].
```

### 2. Build & test commands

The most critical section. An agent that can't build or test is useless.

```markdown
## Commands

\```bash
bun install              # install deps
bun run dev              # start dev server
bun test                 # run tests
bun test --bail          # fail fast
bun run typecheck        # tsc --noEmit
bun run check            # biome lint + format check
\```
```

Include every command a developer needs daily. Add comments for non-obvious flags. If commands differ per package in a monorepo, show the root-level orchestration commands.

### 3. Architecture

Not a design doc — a mental model. Enough for an agent to know where to look.

```markdown
## Architecture

Monorepo with bun workspaces:
- `apps/web/` — Astro marketing site
- `apps/dashboard/` — React SPA (TanStack Router)
- `apis/main/` — Hono API on Cloudflare Workers
- `packages/schema/` — Drizzle ORM, D1 database
- `packages/auth/` — Better Auth config
- `packages/ui/` — Shared React components

API request flow: client → Hono router → middleware (auth, rate limit) → handler → Drizzle → D1
```

For complex systems, describe request flow, key boundaries, and which package owns which concern.

### 4. Conventions

Rules an agent must follow. State them as imperatives with examples.

```markdown
## Conventions

- **Error handling**: return `{ ok, error }` result types — only throw at system boundaries
- **Naming**: camelCase in TS, snake_case in DB columns, kebab-case for files and URLs
- **Commits**: conventional commits (`feat:`, `fix:`, `refactor:`)
- **Testing**: real database in tests, no mocks for the data layer
- **Imports**: use `@scope/package` workspace imports, never relative paths across packages
```

### 5. Project-specific gotchas (optional)

Things a smart agent would get wrong without warning.

```markdown
## Gotchas

- D1 doesn't support `RETURNING *` with `INSERT ... ON CONFLICT` — use separate SELECT
- The auth API must be running locally for dashboard login to work (`bun run dev:apis`)
- Don't edit `src/routeTree.gen.ts` — it's auto-generated by TanStack Router plugin
```

## Length guidelines

| Project size | Target length | Format |
|---|---|---|
| Small (single package, one language) | 50-100 lines | Single `CLAUDE.md` |
| Medium (monorepo, one language) | 100-200 lines | Single `CLAUDE.md` |
| Large (monorepo, multi-language, multi-deploy) | 200-300 lines | `CLAUDE.md` or split to `.claude/rules/` |
| Very large | 30-line `CLAUDE.md` index | `.claude/rules/*.md` by topic |

## Anti-patterns

- **Generic advice** — "Write clean code" / "Follow best practices" wastes tokens
- **Full API docs** — agents can read the code; document patterns, not endpoints
- **Explaining the language** — don't explain what TypeScript generics are
- **Copy-pasting README** — README is for humans browsing GitHub, CLAUDE.md is for agents working in the codebase
- **Stale commands** — a wrong build command is worse than no build command
