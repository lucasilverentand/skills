# Custom hover effects
When appropriate for your application or game, you may implement a custom hover animation that triggers when users view an element. This applies to both system-provided and custom UI elements, including RealityKit entities. A custom hover effect can either substitute or supplement a default visual feedback mechanism.

Before developing custom hover effects, it is crucial to understand their operational mechanics. To enable a custom effect for an element, you must define two distinct states or appearances: one that displays the custom hover animation and another that does not. When a user views the element in your app or game, the system applies this predefined hover effect through a process external to your software. Consequently, you do not receive notifications regarding when the system applies the effect or what state the element is currently in. This out-of-process nature also prevents the custom hover effect from executing code that relies on knowing when a user is viewing the element.

For instance, consider a photo application where the custom effect displays a different symbol based on whether the photo has been favorited. The app specifies the correct symbol for the custom hover effect, and the system renders the effect if the photo is viewed. However, the hover effect cannot perform the favoriting action because the system does not inform the app when viewing occurs.

- **Use a custom hover effect to highlight or improve a specific moment in the user experience.** Users are accustomed to standard hover effects providing visual confirmation or supplementary details (such as in tooltips or tab bars), making a custom effect particularly noticeable. Overusing custom hover effects, or using them when standard ones suffice, can dilute the design's impact, distract users from their primary task, and potentially cause visual unease.
- **Select the appropriate delay.** The custom hover effect can appear immediately, after a brief pause, or after a longer interval, depending on the expected user interaction timeline.
- **No delay (default).** A custom hover effect that appears instantly is particularly effective when the animation is subtle or invites immediate interaction, such as a knob appearing on a slider.
- **Short delay.** Consider a short delay if you want users to be able to quickly interact with an element after viewing it, without waiting for the effect to fully appear; for example, tab bar expansion operates this way.
- **Long delay.** If your custom hover effect presents secondary information, such as a tooltip appearing beneath a button, a slightly extended delay may be beneficial since most users will not need to view this extra information every time.
- **Ensure that one or more of the element’s primary views remain consistent across both states of a custom hover effect.** Maintaining visual stability by keeping at least one primary view unchanged during the animation helps users track the element's transition. If all views of an element shift or change during a custom hover effect, it risks disorienting the user and causing them to lose context.
- **Rigorously test custom hover effects.** Testing is the only way to confirm that a custom hover effect looks polished, responds correctly, and enhances the experience without becoming distracting. Aim to conduct your tests while using Apple Vision Pro to develop an intuitive understanding of how these effects can improve the overall experience.
