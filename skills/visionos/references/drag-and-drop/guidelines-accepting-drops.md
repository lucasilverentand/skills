# Accepting drops
- **Scroll the contents of a destination when necessary.** If a user drags an item within a scrolling container containing extensive content, the system should automatically scroll the destination as the item moves over it. This facilitates precise drop placement; however, if the drag operation continues outside the container boundaries, automatic scrolling is no longer required. This behavior mirrors that of system-provided text views and text fields by default.
- **When there’s a choice, pick the richest version of dropped content your app can accept.** If dragging an object (such as a chart) from another application presents both a rich, native version and a simplified image representation, the app should use the native chart object if supported. Otherwise, it must utilize the image version.
- **Extract only the relevant portion of dropped content if necessary.** For instance, when a contact is dragged to an email recipient field, Mail only displays the name and email address, omitting the contact's full address details.
- **When a physical keyboard is attached, check for the Option key at drop time.** If the user holds the Option key during a drag operation, they can compel a drag-and-drop within the same container to function as a copy. If Option is released before dropping content in that same container, the operation defaults to a move.
- **Provide feedback when dropped content needs time to transfer.** For example, displaying a progress indicator helps users gauge the duration of a lengthy transfer. In collections, lists, and tables, displaying a placeholder at the drop location informs users where to locate the content once transfer is complete. The system may also issue an alert when a time-consuming inter-app transfer occurs.
- **Provide feedback when dropped content initiates a task or action.** If users drop content onto a control that triggers an action (such as printing), the system must indicate that the task has begun and keep the user informed of its status.
- **Apply appropriate styling to dropped text.** If both the source and destination support identical text styles, the dropped text must retain its original attributes (font, typeface, size, etc.). Otherwise, the destination's styling should be applied to the dropped text.
- **After a drop, maintain the content’s selection state in the destination, updating it in the source as needed.** Users expect dropped content to remain selected for immediate action. If the source and destination are the same container: *   A move operation causes the content to disappear from its original location. *   A copy operation within the same container removes the selection state from the content remaining in its original spot. *   When selected content is dragged to a different container, the source content must be deselected.

## Platform guidance — visionOS
If possible, initialize your application to manage content that users drop into an empty area. When you link a user activity to draggable app content, your application gains the ability to open a window or scene dedicated to handling that dropped material. For example, dropping a URL into empty space launches Safari, and Quick Look launches to display content when it is dropped while supported by Quick Look. For developer guidance, consult [NSUserActivity](apple:Foundation/NSUserActivity).
