# Submenus
A menu item may reveal a collection of closely related options within a subordinate list called a *submenu*. A symbol, such as a chevron, placed after the menu item's label indicates that a submenu is present. Functionally, submenus operate identically to standard menus, differing only in their hierarchical placement.

- **Use submenus sparingly.** Each submenu introduces interface complexity and conceals the items it holds. You should consider implementing a submenu when a specific term appears across more than two menu options within the same group. For instance, instead of listing separate items for Sort by Date, Sort by Score, and Sort by Time, a game could use one menu item that employs a submenu to list these sorting choices: Date, Score, and Time. It is generally beneficial to include the repeated term—in this case, *Sort by*—in the menu item's label to help users anticipate the submenu's contents.
- **Limit the depth and length of submenus.** Since revealing multiple levels of hierarchical menus can be difficult for users, it is usually best to restrict them to a single level. Furthermore, if a submenu contains more than approximately five items, consider creating an entirely new menu instead.
- **Ensure a submenu remains accessible even if its nested items are unavailable.** A submenu item, like any menu option, must allow users to open it and view the commands it contains.
- **Prefer using a submenu over indenting menu items.** Indentation is inconsistent with the system and does not clearly communicate the relationships between menu options.
